# 프로젝트 프로세스

> 1인 사이드 프로젝트에 맞는 경량 프로세스.
> 관리 오버헤드를 최소화하면서 "과정의 흔적"을 남기는 것이 핵심.

---

## 0. 방법론

**스크럼(Scrum) 기반 경량 프로세스**를 채택했다.

스크럼의 핵심 요소인 스프린트(2주 단위), 백로그(GitHub Issues), 회고를 가져오되,
1인 프로젝트에 불필요한 요소(데일리 스크럼, 스프린트 플래닝 미팅 등)는 제거했다.

| 스크럼 요소 | 채택 여부 | 이 프로젝트에서의 형태 |
|---|---|---|
| 스프린트 (고정 주기 개발) | ✅ | 2주 단위 |
| 백로그 (작업 목록 관리) | ✅ | GitHub Issues + Milestones |
| 스프린트 회고 | ✅ | `docs/retrospectives/sprint-N.md` (3~5줄) |
| 스프린트 리뷰 (산출물 점검) | ✅ | 프리뷰 배포 확인으로 대체 |
| 데일리 스크럼 | ❌ | 1인이라 불필요 |
| 스크럼 마스터 / PO 역할 분리 | ❌ | 1인이라 불필요 |

### 디자인 패턴

프론트엔드/백엔드의 디자인 패턴은 사전에 확정하지 않는다.
개발 과정에서 복잡도가 증가하여 패턴 도입이 필요해지면, 그 시점에 ADR로 기록한다.

초기 방향성:
- **프론트엔드**: 기능 기반 폴더 구조 + 서버/클라이언트 컴포넌트 경계 분리
- **백엔드 (Route Handlers)**: 파서 / 서비스 / 핸들러 레이어 분리 (관심사 분리 원칙)

## 1. 스프린트 사이클 (2주 단위)

### 스프린트 시작

- 해당 스프린트의 작업을 GitHub Issues로 등록
- 각 이슈에 **"왜 이걸 하는지"** 와 **"완료 조건"** 명시
- GitHub Milestone에 해당 스프린트 이슈들 연결

### 스프린트 진행

- 아래 "기능 단위 작업 플로우"를 반복

### 스프린트 종료

- `docs/retrospectives/sprint-N.md`에 간단한 회고 작성:
  - 이번 스프린트에서 뭘 배웠는가
  - 뭐가 막혔는가
  - 다음에 뭘 바꿀 것인가
- 3~5줄이면 충분. 길게 쓸 필요 없음.

### MVP 후 중간 점검

- Phase 1 MVP 완성 시점에 기획안의 일정/기능 범위가 현실과 맞는지 확인
- 필요 시 Phase 2~3 우선순위 재조정

## 2. 기능 단위 작업 플로우

하나의 기능을 만들 때마다 이 순서를 따른다.

```
1. Issue 생성 (작업 내용 + 완료 조건)
2. 브랜치 생성 (feat/기능명, fix/버그명 등)
3. 구현
4. 테스트 작성
5. PR 생성
6. 셀프 코드 리뷰 (또는 Claude 리뷰)
7. CI 통과 확인
8. 머지
9. 프리뷰 배포 확인
```

### 셀프 코드 리뷰 원칙

- PR을 올린 뒤 최소 몇 시간~하루 뒤에 다시 읽어본다
- 또는 Claude에게 리뷰를 요청한다
- PR에 리뷰 코멘트를 남긴다 (포트폴리오 흔적)

### ADR 작성 시점

매번이 아니라, 아래 상황에서만 작성:

- 기술 스택 선택 (라이브러리, 프레임워크)
- 아키텍처 구조 결정 (상태 관리, 폴더 구조 등)
- 성능 최적화 결정 (측정 결과 기반)
- 트레이드오프가 있는 판단

프로젝트 전체에서 3~5개면 충분.

## 3. 품질 게이트

PR마다 자동으로 체크되는 기준. Phase 0에서 CI를 세팅하면 이후 추가 노력 거의 없음.

### Phase 0~2 기본 게이트

- [ ] ESLint 통과
- [ ] TypeScript 타입 체크 통과
- [ ] Vitest 단위 테스트 통과

### Phase 3 이후 추가 게이트

- [ ] Lighthouse CI 점수 확인 (성능 저하 감지)
- [ ] 번들 사이즈 변화 확인
- [ ] Playwright E2E 테스트 통과

**원칙**: 게이트를 통과 못 하면 머지하지 않는다.

## 4. 브랜치 전략

```
main          ← 배포 브랜치 (Vercel 자동 배포)
  └── feat/*  ← 기능 개발
  └── fix/*   ← 버그 수정
  └── docs/*  ← 문서 작업
  └── chore/* ← 설정, 의존성 등
```

- main에 직접 push 금지. 항상 PR을 통해 머지.
- 브랜치 이름 예시: `feat/crawling-parser`, `fix/boardid-gap`, `docs/adr-001`

## 5. 커밋 컨벤션

```
<type>: <description>

예시:
feat: 메인 페이지 HTML 파서 구현
fix: boardId 연속성 체크에서 빈 번호 누락 수정
test: 크롤링 파서 단위 테스트 추가
docs: ADR 001 기술 스택 선정 근거 작성
chore: ESLint 규칙 업데이트
style: Tailwind 클래스 정리
refactor: 공고 타입 정의 구조 변경
```

### type 목록

| type | 용도 |
|---|---|
| feat | 새 기능 |
| fix | 버그 수정 |
| test | 테스트 추가/수정 |
| docs | 문서 |
| chore | 설정, 빌드, 의존성 |
| style | 코드 포맷팅 (기능 변화 없음) |
| refactor | 리팩토링 (기능 변화 없음) |
| perf | 성능 개선 |

## 6. Issue / PR 템플릿

### Issue

```markdown
## 작업 내용
[무엇을 할 것인지]

## 왜?
[이 작업이 필요한 이유]

## 완료 조건
- [ ] [구체적 조건 1]
- [ ] [구체적 조건 2]
- [ ] [관련 테스트 작성]
```

### PR

```markdown
## 변경 사항
[무엇을 변경했는지]

## 관련 이슈
closes #[이슈 번호]

## 테스트
- [ ] 단위 테스트 추가/수정
- [ ] E2E 테스트 추가/수정 (해당 시)
- [ ] 수동 테스트 완료

## 스크린샷 (UI 변경 시)
```
