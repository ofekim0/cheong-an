# 청안 — 프로젝트 기획안

> **프로젝트명**: 청안 (cheong-an) — 청년안심주택 새 공고 알림 서비스
> **목적**: 네카라쿠배 이직용 포트폴리오 (1순위) + 본인 실사용 서비스
> **작성일**: 2026-03-19
> **상태**: Phase 0 진행 중

---

## 1. 프로젝트 개요

### 1-1. 문제 정의

청년안심주택 홈페이지(https://soco.seoul.go.kr/youth/main/main.do)에는
새 모집공고 알림 기능이 없다.
수요자는 공고가 언제 올라올지 모르기 때문에 수시로 직접 접속해서 확인해야 하며,
이를 놓치면 청약 기회를 잃게 된다.

### 1-2. 해결 방안

모집공고 페이지를 주기적으로 모니터링하여 새 공고(최초 모집 / 추가 모집)를 감지하고,
사용자에게 즉시 알림을 보내는 서비스를 만든다.

### 1-3. 데이터 소스 및 크롤링 전략

**데이터 소스**: 청년안심주택 홈페이지 모집공고

**크롤링 가능성 조사 결과** (2026-03-19 확인 완료):

| 페이지                             | 렌더링 방식                         | 크롤링 가능 여부                 |
| ---------------------------------- | ----------------------------------- | -------------------------------- |
| 메인 페이지 (main.do)              | SSR — 최신 5건 HTML 포함            | ✅ 단순 HTTP GET으로 가능        |
| 모집공고 리스트 (list.do)          | CSR — JS 동적 로딩 (테이블 빈 상태) | ⚠️ AJAX 엔드포인트 파악 필요     |
| 개별 공고 상세 (view.do?boardId=N) | SSR — 전체 내용 HTML 포함           | ✅ boardId만 알면 바로 접근 가능 |

**채택 전략**: 메인 페이지 모니터링 + boardId 연속성 보완

1. 메인 페이지를 주기적으로 요청하여 최신 5건의 boardId 추출
2. 마지막 확인 boardId와 비교하여 새 공고 감지
3. 5건 초과 동시 등록 대비: boardId 범위 내 빈 번호의 상세 페이지를 직접 fetch하여 누락 방지
4. 새 공고 감지 시 상세 페이지 크롤링 → 정보 저장 → 알림 발송

**법적 리스크**: 공개된 공공기관 웹페이지에 대한 저빈도(1시간 간격) 확인. 문제 없음.

### 1-4. "왜" 3층 검증

| 층  | 질문                       | 답변                                                                     | 상태 |
| --- | -------------------------- | ------------------------------------------------------------------------ | ---- |
| 1층 | 왜 만들었는가              | 본인이 청년안심주택 실수요자. 알림 없어서 매번 직접 확인해야 하는 불편함 | ✅   |
| 2층 | 기존 서비스와 뭐가 다른가  | 아래 기존 서비스 조사 참고                                               | ✅   |
| 3층 | 만드는 것이 왜 의미 있는가 | 개인 동기 + 프론트엔드 엔지니어링 깊이 증명                              | ✅   |

### 1-5. 기존 서비스 조사 (2026-03-19 완료)

| 서비스             | 청년안심주택 알림                                                 | 한계                                             |
| ------------------ | ----------------------------------------------------------------- | ------------------------------------------------ |
| 마이홈 앱          | 임대주택 모집공고 알림 있으나 청년안심주택이 별도 카테고리로 없음 | 지역 필터가 시/도 단위까지만 가능 (구 단위 불가) |
| SH 청약알리미      | 장기전세, 국민임대, 장기안심만 대상                               | 청년안심주택 민간임대 추가모집 커버 안됨         |
| 청약홈             | APT 분양 위주                                                     | 청년안심주택은 별도 관리                         |
| 공공주택 알리미 앱 | LH/SH/GH 공고 지도 표시                                           | 청년안심주택 전용 아님                           |

**결론**: 청년안심주택 전용 알림 서비스는 현재 존재하지 않음.

**차별화 포인트**: 청년안심주택 전용 + 서울시 구 단위 필터 + 최초/추가/민간/공공 구분 알림

## 3. 기술 스택 설계

### 3-1. 프론트엔드

| 기술                     | 선택 근거 (ADR)                                                                                       |
| ------------------------ | ----------------------------------------------------------------------------------------------------- |
| **Next.js (App Router)** | 얼마니에서 Pages Router 사용 → App Router 전환. RSC, 병렬 라우트 학습. SSG/ISR로 공고 목록 정적 생성. |
| **TypeScript**           | 크롤링 데이터 파싱 시 타입 안전성, API 응답 타입 정의                                                 |
| **Zustand**              | 알림 설정 상태 관리. persist middleware, selector 최적화 등 심화 패턴 적용                            |
| **Tailwind CSS**         | Emotion → Tailwind 전환. 번들 사이즈 최적화 근거 확보 (런타임 vs 빌드타임 비교)                       |
| **TanStack Query**       | 공고 데이터 캐싱, 백그라운드 리프레시                                                                 |

### 3-2. 백엔드 / 인프라

| 기술                                        | 선택 근거                                               |
| ------------------------------------------- | ------------------------------------------------------- |
| **Next.js Route Handlers**                  | 별도 백엔드 서버 없이 크롤링 트리거, 알림 발송 API 구현 |
| **Cron Job (Vercel Cron / GitHub Actions)** | 주기적 크롤링 스케줄링                                  |
| **Supabase**                                | 공고 데이터 저장, 사용자 알림 설정 저장                 |
| **Vercel**                                  | 배포 + CI/CD + Edge Functions                           |

### 3-3. 테스트

| 도구           | 용도                                                      |
| -------------- | --------------------------------------------------------- |
| **Vitest**     | 단위 테스트 — 크롤링 파서, boardId 감지 로직, 필터링 로직 |
| **Playwright** | E2E 테스트 — 알림 설정 플로우, 공고 조회 플로우           |
| **MSW**        | API 모킹 — 크롤링 응답 시뮬레이션                         |

### 3-4. 기타

| 도구               | 용도                                               |
| ------------------ | -------------------------------------------------- |
| **GitHub Actions** | CI/CD 파이프라인 — lint, 테스트, 빌드, 배포 자동화 |
| **Lighthouse CI**  | 성능 측정 리포트 자동 생성                         |
| **Sentry**         | 에러 모니터링 — 크롤링 실패, 알림 발송 실패 감지   |

### 3-5. 얼마니 업무와의 기술적 차별화

| 얼마니                     | 이 프로젝트                                   |
| -------------------------- | --------------------------------------------- |
| Pages Router               | App Router (RSC, 병렬 라우트)                 |
| Emotion (런타임 CSS-in-JS) | Tailwind (빌드타임 유틸리티)                  |
| 폼 / 테이블 / 필터링 UI    | 크롤링 파이프라인 + 알림 시스템 + 시각화      |
| 프론트엔드 Only            | Route Handlers로 백엔드 로직 직접 구현        |
| 테스트 코드 없음           | Vitest + Playwright + MSW                     |
| 수동 배포                  | GitHub Actions + Vercel CI/CD + Lighthouse CI |

### 면접 대비 예상 Q&A

| 예상 질문                    | 답변 방향                                                                              |
| ---------------------------- | -------------------------------------------------------------------------------------- |
| 왜 이 프로젝트를 만들었나요? | 본인이 실수요자. 알림 기능이 없어서 직접 만들었다.                                     |
| 왜 App Router?               | 회사에서 Pages Router 사용 → 차이를 체감하고 싶었다. RSC가 적합한 이유. ADR 참고.      |
| 왜 Tailwind?                 | Emotion 대비 번들 사이즈 감소. 빌드타임 vs 런타임 성능 차이 측정 결과. ADR 참고.       |
| 테스트 전략은?               | 크롤링 파서는 HTML 스냅샷 기반 단위 테스트, 알림 플로우는 E2E. MSW로 외부 의존성 격리. |
| 성능 최적화 경험?            | Lighthouse CI 히스토리로 LCP 개선 과정 설명. 번들 분석 → 다이나믹 임포트 적용.         |
